# Revised High-Level Architecture: PreMortem AI

## Goal

PreMortem AI should not be limited to one MRI procurement example. It should become a generic agentic business decision review framework. (Continuous monitoring could be focus if time permits.)

The MRI procurement scenario can remain the first demo use case, but the architecture should support few other workflows later.

## Core Idea

PreMortem AI simulates a business review board before a high-risk decision is approved.

It uses multiple agents that work in parallel, evaluate different dimensions of risk, and return structured findings to an orchestrator.

## Main Components

### 1. Workflow Input Layer

Accepts a business decision request.

For the current demo, the input is MRI procurement data.

Later, the same structure can support:

- Vendor evaluation
- Contract review
- Cost optimization
- Procurement approval
- Compliance review

### 2. Orchestrator Agent

The orchestrator is the manager of the workflow.

Responsibilities:

- Receive business input
- Decide which agents should run
- Pass the right data to each agent
- Run agents in parallel where possible
- Wait for agent responses
- Collect agent outputs
- Send results to evaluator
- Prepare final response for UI

For the next two days, the orchestrator can be implemented as a dummy/controller layer that calls existing agents and returns structured intermediate states.

### 3. Specialist Agents

These agents should be more generic than the current MRI-specific interpretation.

Suggested minimal agents for demo:

#### Contract / Terms Agent

Checks payment terms, warranty clauses, penalties, liability, and commercial risks.

#### Cost / Benchmark Agent

Checks whether quoted cost appears reasonable compared with available benchmarks, historical purchases, or expected market range.

#### Operational Readiness Agent (Chinmaey>> I think this addresses the idea of updating checklists procedure etc.)

Checks whether the business is ready to execute the decision after approval.

For MRI, this includes site readiness, approvals, electrical readiness, technicians, and training.

#### Vendor / External Risk Agent (Chinmaey>> Does this represent the competetor research?)

Checks vendor performance, reputation, delivery history, and external risks where data is available.

### 4. Evaluator Agent

The evaluator reviews the overall quality of the workflow.

Responsibilities:

- Check if all required agents completed
- Check if agent outputs are consistent
- Identify missing evidence
- Identify conflicting recommendations
- Estimate confidence
- Decide whether final output is ready for approval review

This is important because it shows that the system is not just generating text, but monitoring quality.

### 5. Decision Output Layer

Final output should include:

- Go / No-Go / Conditional Go
- Overall risk score
- Key risks
- Supporting evidence
- Missing information
- Recommended next actions
- Agent-level status
- Confidence score

### 6. UI Direction

The UI should show the agentic workflow, not just the final answer.

Suggested UI sections:

- Input summary
- Orchestrator status
- Agent execution cards
- Agent findings
- Evaluator feedback
- Final recommendation
- Future workflow placeholders

Even if all workflows are not implemented, the UI can show architecture readiness.

## Why This Better Addresses Agentic AI

The revised architecture demonstrates:

- Parallel agent execution
- Orchestration
- Role-based agents
- Evaluation and quality monitoring
- Structured decision output
- Expandable workflow design

This is stronger than a single-agent analysis demo.
